# アーキテクチャ

## 設計原則

- **Markdownベース**: LLMの生成・消費に最適。テンプレートエンジン不要
- **ファイルシステム=データベース**: Gitがバージョニング、Markdownが構造を提供。過度な構造化を避ける
- **symlink不使用**: Windows対応のため明示的ファイル生成方式
- **スキルは宣言的**: WHAT（何をするか）を記述し、HOW（実行方法）はAIが判断
- **多様性保持**: 複数アプローチのpros/consを残す
- **git worktreeによる並列作業**: Claude/Codex計画の並列策定、エージェント間のファイル競合防止にworktreeを活用

## データフロー

### ハーネス生成
1. specs/ から最新仕様を参照
2. templates/ から目的別テンプレートを選択
3. 対象プロジェクトのコードベースを分析
4. テンプレート＋分析結果からハーネスを生成

### クロスプラットフォーム同期
1. ソースハーネスを読み取り
2. mapping/ の変換ルールを適用
3. ターゲットハーネスを生成/更新
4. 非対称機能（skills, hooks等）はドキュメントで代替

### 自律改善サイクル
1. セッション終了時に自己評価
2. 巡回で外部更新をキャッチアップ
3. 蓄積した評価・更新から改善計画を策定
4. 改善実行（自律レベルに応じてユーザー承認）
5. ADRに判断を記録
