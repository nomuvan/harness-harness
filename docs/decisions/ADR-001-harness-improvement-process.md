# ADR-001: ハーネス改善の作業プロセス

## ステータス
承認

## コンテキスト
project-alphaへの初回ハーネス改善適用で確立した作業プロセスを標準化する必要がある。

## 決定

対象プロジェクトのハーネス改善は以下のプロセスで実施する：

### 1. 診断
- 対象プロジェクトのハーネスファイルを全て読み取り
- specs/の最新仕様と突き合わせて未活用機能を特定
- ドメイン非依存パターンとドメイン固有を分離
- 診断レポートを `logs/evaluations/` に出力

### 2. 改善提案
- 優先度A-D で分類した改善提案書を作成
- A: 即効改善（低リスク・高ROI）
- B: 新機能キャッチアップ
- C: 構造改善
- D: 次ステップの布石
- ユーザーと議論して取捨選択

### 3. 実装
- 対象プロジェクトでgit worktreeを作成
- 改善を一括実装
- ソースコードは変更しない（ハーネスファイルのみ）

### 4. PR・レビュー・マージ
- PRを作成（変更の背景にharness-harnessからの診断である旨を記載）
- GitHub ActionsのCIレビュー指摘に対応（取捨選択含む）
- マージ後worktreeを削除

### 5. フィードバック
- 改善過程で得た知見をharness-harnessにフィードバック
- ドメイン非依存パターンをtemplates/の候補として記録
- プロジェクト間コピー可能なskill/rules/hooksを特定

## 理由
- project-alphaでの初回適用で有効性が確認された
- worktree使用でメインブランチを汚さない
- PR経由でCIレビューを活用できる
- プロセス自体をharness-harnessのスキル化する布石

## 影響
- 今後の全プロジェクトへのハーネス改善はこのプロセスに従う
- `/diagnose-harness` スキルの仕様の元になる
